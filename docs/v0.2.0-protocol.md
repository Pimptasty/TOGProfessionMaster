# TOG Profession Master — v0.2.0 Sync Protocol

**Status:** Design (pre-implementation)
**Supersedes:** v0.1.x full-payload broadcast model
**Requires:** DeltaSync-1.0 MINOR ≥ 9 (shipped in DeltaSync v2.0.3)

---

## 1. Why this exists

The v0.1.x protocol broadcasts each peer's full profession + cooldown payload to the guild every 30 seconds. For a 50-character guild that's roughly 30 KB × 50 broadcasters = 1.5 MB of guild-channel traffic per minute, even when nothing has changed. It also doesn't relay — if Alice is offline, no one can fill in her cooldowns or recipes for newly-arriving guildmates.

v0.2.0 replaces this with a TOGBank-style hash-then-fetch model:

- Each peer broadcasts a tiny **hash list** every ~10 minutes (or on local change).
- Peers compare hashes; when they differ, they whisper a small handshake.
- The chosen sender broadcasts the differing **leaf data** on guild — not via whisper — so every peer with stale data for that leaf merges for free.
- Data merge is **content-aware** so receiving from any peer always converges to the right state.
- Anyone with cached data for any character can serve it (relay), so cooldown / recipe coverage no longer requires the owner to be online.

The result: steady-state traffic is near-zero, and active churn produces ~1 KB whisper handshakes plus small targeted guild broadcasts.

## 2. Channel allocation

| Channel | Distribution | Priority | Purpose | Size per message |
| ------- | ------------ | -------- | ------- | ---------------- |
| `OFFER` (hash list) | GUILD | BULK | L0 hash broadcast (every 10 min, or on local change) | ~600 B per broadcaster |
| `OFFER` (hash offer) | WHISPER | NORMAL | Peer responds "I have leaves X, Y, Z" | ~30 B per offer |
| `QUERY` (request) | WHISPER | NORMAL | Broadcaster asks chosen sender "send me leaf X" | ~30 B per request |
| `DATA` (leaf payload) | GUILD | BULK | Per-leaf content broadcast on guild — every peer can merge | 100 B – 5 KB per leaf |

Whispers are reserved for handshake control messages only. **All bulk data flows on the guild channel.** Whisper throttling makes per-recipient data transfer infeasible at scale.

`SendData` (which uses WHISPER `RESPONSE`) is unused by TOGPM in v0.2.0. The library API stays available for other DeltaSync consumers.

---

## 3. Hash leaf taxonomy

Each leaf is identified by a string `itemKey` and stored in `gdb.hashes[itemKey] = { hash, updatedAt }`.

| Leaf | `itemKey` | Content hashed | Change frequency |
| ---- | --------- | -------------- | ---------------- |
| Recipe metadata, per profession | `recipemeta:<profId>` | All immutable recipe fields (name, icon, isSpell, spellId, itemLink, recipeLink, reagents) for every recipe in that profession | Bootstrap-only — never changes after first guild sync |
| Crafter membership, per profession | `crafters:<profId>` | `{ recipeId → { charKey → true } }` — who knows what | Per learn/unlearn |
| Cooldowns, per character | `cooldown:<charKey>` | Whole cooldown bucket `{ spellId → expiresAt }` for one character | Per cast |
| Account characters, per broadcaster | `accountchars:<charKey>` | Alt group owned by that broadcaster `{ charKey → true }` | Per new alt creation (rare) |
| Cooldown roll-up | `guild:cooldowns` | XOR over all `cooldown:<charKey>` hashes | Whenever any per-character cooldown leaf changes |
| Account characters roll-up | `guild:accountchars` | XOR over all `accountchars:<charKey>` hashes | Whenever any per-broadcaster alt leaf changes |

### What gets broadcast at L0

Each peer's hash list broadcast contains:

- `recipemeta:<profId>` × 9 (one per profession the guild has data for)
- `crafters:<profId>` × 9
- `guild:cooldowns` × 1 (roll-up)
- `guild:accountchars` × 1 (roll-up)

**Total: ~20 hash entries per peer per broadcast.** At ~30 B per entry that's ~600 B — fits in one or two AceComm chunks.

Per-character `cooldown:<charKey>` and `accountchars:<charKey>` leaves are **NOT** in the L0 broadcast. They're requested via L1 drill-down on roll-up mismatch.

### Why roll-ups for per-character leaves but not for per-profession

A 300–500 character guild has ~300–500 `cooldown:<charKey>` leaves and ~50 `accountchars:<charKey>` leaves. Broadcasting all per-character hashes every 10 min would be ~600–1000 hashes per peer × 50 broadcasters = unacceptable wire volume.

Per-profession recipe leaves max out at 9 (9 professions). Cheap enough to broadcast directly without roll-up.

## 4. The hash + timestamp invariant

Each leaf entry is `{ hash, updatedAt }` where:

- `hash` is a numeric checksum of the leaf's content. Pure function of content.
- `updatedAt` is the **content-derived** timestamp — when the underlying data last changed.

**Invariant:** hash and updatedAt are co-determined by the data state. Both change atomically when content changes, both stay frozen otherwise.

### Implications

- Two peers with identical content for a leaf have identical `(hash, updatedAt)` tuples. They both got the data from the same authoritative source (a scan), and the tuple was computed once at scan time and propagated unchanged.
- A peer receiving a leaf payload stores the incoming `(hash, updatedAt)` **verbatim**. No recomputation.
- A peer merging conflicting payloads (content-aware merge produces a new state different from either input) computes a new `(hash, updatedAt)` from the merged state. `updatedAt = max(input timestamps)` since the merged state is at least as fresh as the freshest input.
- The current `HashManager:RebuildAll` violates this invariant by writing `updatedAt = GetServerTime()` on every call regardless of content change. **It must be retired in v0.2.0.**

### Content-derived `updatedAt` per leaf type

| Leaf | `updatedAt` formula |
| ---- | ------------------- |
| `cooldown:<charKey>` | `gdb.lastScan[charKey].cooldowns` (server time when that character last had its cooldowns scanned, locally or relayed) |
| `crafters:<profId>` | `max(gdb.lastScan[crafterKey][profId])` across all crafters in that profession |
| `recipemeta:<profId>` | Same as `crafters:<profId>` — when the most recent contributing scan happened |
| `accountchars:<charKey>` | `gdb.lastScan[charKey].accountchars` |
| `guild:cooldowns` | `max(updatedAt)` across all `cooldown:<charKey>` leaves |
| `guild:accountchars` | `max(updatedAt)` across all `accountchars:<charKey>` leaves |

`gdb.lastScan` is a new top-level field (see §8).

### Use of `updatedAt`

- **Skip-on-match:** if my hash for a leaf equals the broadcaster's, no transfer (skip-on-match also implies same `updatedAt` under the invariant; the protocol checks hash equality only).
- **Routing:** when multiple peers offer to serve a leaf, the broadcaster picks the offer with the highest `updatedAt`. With content-derived timestamps this is "the freshest copy on the network."

DeltaSync MINOR 9's `OnHashListReceived` triggers offers on hash-mismatch (regardless of `updatedAt`). The candidate-collection sort in `OnOffer` is by `updatedAt` descending. Both stay as-is — TOGPM just needs to populate `updatedAt` correctly.

## 5. Wire formats

### 5.1 L0 hash list broadcast

Sent via `DS:BroadcastItemHashes(items, "BULK")` → GUILD:

```lua
{
    type  = "hash-list",
    items = {
        ["recipemeta:171"]   = { hash = 12345, updatedAt = 1714400000 },
        ["recipemeta:202"]   = { hash = 67890, updatedAt = 1714400000 },
        ...
        ["crafters:171"]     = { hash = 11111, updatedAt = 1714450000 },
        ...
        ["guild:cooldowns"]  = { hash = 22222, updatedAt = 1714460000 },
        ["guild:accountchars"] = { hash = 33333, updatedAt = 1714400000 },
    },
}
```

### 5.2 Hash offer (whisper)

Sent via `DS:SendHashOffer(target, items)` → WHISPER:

```lua
{
    type  = "hash-offer",
    items = {
        ["crafters:171"] = { hash = 99999, updatedAt = 1714470000 },
        -- only the leaves where my hash differs from the broadcaster's
    },
}
```

### 5.3 Roll-up drill-down request (whisper)

When the broadcaster's `guild:cooldowns` differs from ours, we ask the broadcaster for their per-character cooldown hashes:

Sent via `DS:RequestData(target, baseline)` → WHISPER:

```lua
baseline = {
    type    = "subhashes",
    parent  = "guild:cooldowns",   -- or "guild:accountchars"
}
```

The responder builds a hash list of all `cooldown:<charKey>` leaves they have and broadcasts that on guild as a leaf payload (see §5.5) with `type = "hash-list-response"`. Receivers compare per-character hashes locally and identify which characters' data they need to request.

### 5.4 Leaf data request (whisper)

After identifying a specific differing leaf, ask the chosen sender to broadcast it:

Sent via `DS:RequestData(target, baseline)` → WHISPER:

```lua
baseline = {
    type  = "leaf-data",
    keys  = { "cooldown:Alice-Realm" },   -- one or more leaves
}
```

### 5.5 Leaf data broadcast

The chosen sender responds by broadcasting the leaf content on guild via `DS:BroadcastData(payload, "GUILD", "BULK")`:

```lua
{
    type      = "leaf-data",
    sender    = "Bob-Realm",
    timestamp = 1714470000,
    leaves = {
        ["cooldown:Alice-Realm"] = {
            data       = { [17187] = 86400, [11479] = 0 },   -- relative seconds remaining
            hash       = 99999,
            updatedAt  = 1714470000,
        },
    },
    -- Optional: lastScan info for any charKeys touched by this payload
    lastScan = {
        ["Alice-Realm"] = { cooldowns = 1714470000 },
    },
}
```

Multiple leaves can ride in one broadcast. The receiver's `OnGuildDataReceived` iterates `payload.leaves` and merges each.

## 6. Message flow / sequence

### 6.1 Steady-state quiet guild

```text
Alice ──── BroadcastItemHashes (GUILD) ────►  All peers
All peers compare; all hashes match; no offers; nothing to do.
```

Wire cost per 10-min cycle: ~600 B per broadcaster on guild, ~30 KB total for 50 broadcasters. Zero whispers, zero data transfer.

### 6.2 Alice casts a transmute

```text
1. Alice's local scan updates gdb.cooldowns[Alice].
   Targeted invalidation: cooldown:Alice hash + updatedAt change.
   guild:cooldowns roll-up hash + updatedAt also recompute.

2. Within 10 min, Alice broadcasts hash list (GUILD).
   guild:cooldowns hash differs from peers' caches.

3. Peers see mismatch on guild:cooldowns, whisper hash-offer back to Alice (WHISPER).
   ("Here's my hash for guild:cooldowns" — peers don't yet know which character changed.)

4. Alice's collect window expires.  Alice picks the offer with the highest updatedAt
   (or her own if hers is highest — in this case it is; Alice is the source).

5. Alice's hash IS the freshest, so peers need to fetch FROM her, not the other way around.
   Each peer who saw the mismatch whispers Alice: "send me subhashes for guild:cooldowns".

6. Alice broadcasts a hash-list-response on GUILD containing per-character cooldown
   hashes for everyone Alice knows about (~50 entries).

7. Each peer compares per-character hashes locally.  Identifies cooldown:Alice differs
   (or possibly some other peer too if their data drifted).

8. Each peer whispers Alice: "send me cooldown:Alice".

9. Alice broadcasts leaf-data for cooldown:Alice on GUILD.  All 50 peers merge from the
   single broadcast.
```

In practice peers can batch the requests (one whisper for multiple leaves) and Alice batches responses (one broadcast carrying multiple leaves). This isn't separately specified — `baseline.keys` is an array, payload `leaves` is a map.

### 6.3 New peer logs in

Carol logs in fresh. Her gdb is empty:

```text
1. Carol broadcasts hash list (GUILD) with empty/zero hashes.

2. Other peers see Carol's hash for every leaf differs.  Each peer with content
   whispers offers for the leaves they have.

3. Carol picks one offer per leaf (sorted by updatedAt desc, least-loaded peer
   on ties — DeltaSync's existing PickPeer logic).

4. Carol whispers each chosen sender for the missing leaves.

5. Senders broadcast leaf data on GUILD.  Carol merges everything in.
```

For a fully-empty Carol joining a fully-populated guild: ~20 leaves × ~500 B = ~10 KB on guild plus ~20 whispers. Done in ~30 s.

## 7. Merge rules per data type

When `OnGuildDataReceived` processes an incoming `payload.leaves` map, each leaf entry merges according to its type. The merge rules below assume content-aware (CRDT-style) merge.

### 7.1 Cooldowns (`cooldown:<charKey>`)

Per `(charKey, spellId)`:

```lua
gdb.cooldowns[charKey][spellId] = max(local.expiresAt or 0, incoming.expiresAt)
```

`expiresAt` only ever moves forward when a cast happens. `max` always picks the most-recent cast. Stale relayer data is a no-op (their value is ≤ ours).

If incoming has `expiresAt = 0` (Ready) and local has a future timestamp, `max` keeps the future timestamp — correct, the future one is fresher.

### 7.2 Crafters (`crafters:<profId>`)

Per `(profId, recipeId)`:

- **Authoritative replace** for the broadcaster's own scan: if `data.charKey == sender's charKey` and the sender's `lastScan[sender][profId]` is fresh, wipe `crafters[recipeId][sender]` from every existing recipe in that profession before re-applying the incoming set. Handles the "Alice unlearned recipe X" case.
- **Union merge** for relayed data: for each `(recipeId, charKey)` in the incoming crafters map, set `gdb.recipes[profId][recipeId].crafters[charKey] = true`. Never remove via relay.

### 7.3 Recipe metadata (`recipemeta:<profId>`)

Per recipe:

- If local has no entry, store incoming verbatim.
- If local has an entry, prefer the **richest non-nil** version per field:
  - `name`, `icon`, `isSpell`, `spellId`: take incoming if local is nil.
  - `itemLink`, `recipeLink`: take incoming if local is nil or empty string.
  - `reagents`: per-reagent merge via existing `mergeReagents` helper (already in `Scanner.lua` from v0.1.5) — preserves itemId / itemLink per reagent when one side is missing them.

Recipe metadata is treated as **immutable in steady state** — the merge rules above only matter during initial sync or when one peer has data another doesn't. Once everyone has the same metadata for a recipe, hashes match and no more merging happens.

### 7.4 Account characters (`accountchars:<charKey>`)

Per broadcaster:

- **Authoritative replace** when incoming data has `data.charKey == broadcasterKey` (the broadcaster is asserting their own group).
- **Union merge** otherwise (relay): never remove members via relay.

```lua
-- Authoritative path (broadcaster = data.charKey)
gdb.accountChars[broadcasterKey] = { ...incoming members... }

-- Relay path
for _, member in ipairs(incoming) do
    add member to gdb.accountChars[broadcasterKey]   -- union
end
```

After updating `gdb.accountChars`, rebuild the derived `gdb.altGroups` denormalized lookup table.

### 7.5 `lastScan` timestamps

Per `(charKey, scope)` where scope is a profId or `"cooldowns"` or `"accountchars"`:

```lua
gdb.lastScan[charKey][scope] = max(local, incoming)
```

`lastScan` only goes up. Pure max merge, no exceptions.

## 8. Storage layout

### 8.1 New `gdb` fields

```lua
gdb.accountChars = {
    -- Per-broadcaster authoritative alt group.  The denormalized
    -- gdb.altGroups view is rebuilt from this whenever it changes.
    ["Bob-Realm"]   = { "Bob-Realm", "Robert-Realm", "Bobby-Realm" },
    ["Alice-Realm"] = { "Alice-Realm", "Alyssa-Realm" },
}

gdb.lastScan = {
    -- When the most recent authoritative scan happened for each
    -- (character, scope).  Scope is a profId, "cooldowns", or
    -- "accountchars".  Used to compute content-derived updatedAt.
    ["Alice-Realm"] = {
        [171]            = 1714400000,   -- alchemy
        [202]            = 1714350000,   -- engineering
        cooldowns        = 1714400500,
        accountchars     = 1714350000,
    },
}
```

### 8.2 Fields whose shape doesn't change

- `gdb.recipes[profId][recipeId] = { name, icon, ..., crafters = { [charKey] = true } }` — same as v0.1.5.
- `gdb.cooldowns[charKey] = { [spellId] = expiresAt }` — same as v0.1.5.
- `gdb.skills[charKey][profId] = { skillRank, skillMax }` — same as v0.1.5.
- `gdb.specializations[charKey] = { [profId] = spellId }` — same as v0.1.5.
- `gdb.altGroups[charKey] = { ...members }` — same shape, but rebuilt from `gdb.accountChars` instead of merged directly.
- `gdb.guildData[charKey] = {}` — same as v0.1.5 (membership marker).
- `gdb.hashes[itemKey] = { hash, updatedAt }` — same shape; new leaf keys (see §3); content-derived `updatedAt` semantics (see §4).

### 8.3 Migration from v0.1.5

On first v0.2.0 load:

1. `gdb.accountChars = {}` if missing. Populate as v0.2.0 broadcasts arrive. Existing `gdb.altGroups` keeps working from already-merged data; new updates flow through `gdb.accountChars`.
2. `gdb.lastScan = {}` if missing. Populate on next scan/receive. Until populated, fall back to `GetServerTime()` for `updatedAt` (one-time approximation; converges within a scan cycle).
3. `gdb.hashes`: nuke or leave. The new leaf keys (`recipemeta:<profId>`, `crafters:<profId>`, `accountchars:<charKey>`, `guild:cooldowns`, `guild:accountchars`) don't exist yet, so they get computed fresh on the first invalidation. Old keys (`recipes:<profId>`) become unused garbage; can be deleted in a subsequent version.

No existing data is destroyed. Existing recipes, cooldowns, alt groups all remain usable.

### 8.4 DeltaSync namespace bump

To prevent v0.1.5 ↔ v0.2.0 cross-talk corrupting either side's parsers:

```lua
DS:Initialize({
    namespace = "TOGPmv2",   -- was "TOGPmv1"
    ...
})
```

v0.1.5 peers with namespace `TOGPmv1` and v0.2.0 peers with `TOGPmv2` won't see each other's broadcasts. Each version operates in isolation. Once the guild has fully upgraded, the v0.1.5 namespace is dead.

## 9. Implementation work breakdown

Order of work, smallest blast radius first:

1. **Storage migration helpers.** Initialize `gdb.accountChars`, `gdb.lastScan`. One-time on first v0.2.0 load.
2. **`gdb.lastScan` population.** Stamp from every scan path (`ScanTradeSkillInto`, `ScanCraftSkillInto`, `ScanCooldowns`, account-char registration in `OnPlayerEnteringWorld`).
3. **`HashManager` overhaul.**
   - Delete `RebuildAll` (or rename and call only on first-load when `gdb.hashes` is empty).
   - Add new leaf keys: `recipemeta:<profId>`, `crafters:<profId>`, `accountchars:<charKey>`, `guild:cooldowns`, `guild:accountchars`.
   - `setEntry` semantics: `updatedAt` parameter is content-derived; callers must compute it from `gdb.lastScan` or content, never `GetServerTime()`.
   - `setEntry` no-op guard: if `hash == existing.hash and updatedAt == existing.updatedAt`, skip the write entirely.
4. **Targeted invalidation in scan paths.** Each scan calls only the `Invalidate*` helpers for what it touched. Drop the unconditional `RebuildAll` call from `OnGuildDataReceived`.
5. **Differential broadcast.** `Scanner:BroadcastHashes` sends only leaves that have changed since the last broadcast (tracked in a per-session `_lastBroadcastHashes` cache). First broadcast after login = full list; subsequent broadcasts = deltas only.
6. **`HasContent` opens up to relay.** Returns true for any cached leaf, not just owner-owned.
7. **Wire format changes.** New `BuildLeafPayload(itemKey)` produces per-leaf data. `BroadcastData` replaces `SendData` for leaf responses.
8. **`OnGuildDataReceived` rewrite.** Content-aware merge per data type per §7. Returns/tracks dirty leaves; only those get re-hashed.
9. **Drill-down request handlers.** `onDataRequest(sender, baseline)` reads `baseline.type`:
   - `"subhashes"` → respond with hash list for the requested parent's children, broadcast on GUILD.
   - `"leaf-data"` → respond with leaf payload for `baseline.keys`, broadcast on GUILD.
10. **DeltaSync MINOR ≥ 9 guard.** In `Scanner:InitDeltaSync`, refuse to enable sync if MINOR < 9. Print a clear message.
11. **Namespace bump** `TOGPmv1` → `TOGPmv2`.
12. **Diagnostic commands.** `/togpm dumphashes`, `/togpm dumpcooldowns <charKey>`, `/togpm forcebroadcast` (bypasses 10-min debounce). Useful for testing.

## 10. Out of scope for v0.2.0

- **Per-spell cooldown hashes.** Per-character bucket transfer is small enough; per-spell adds complexity for marginal wire savings.
- **Hash + timestamp packed into one value.** Conceptually clean but no real wire-size win at our scale; keeping `{hash, updatedAt}` as separate fields aligns with TOGBank's proven shape.
- **Static-data signing / corruption detection.** If two peers compute different `recipemeta:<profId>` hashes for what should be identical Blizzard recipe data, that's a corruption signal worth logging — but acting on it (auto-quarantine, manual rebuild) is future work.
- **Cross-guild relay.** TOGPM stays guild-scoped. Confederate guild support (GreenWall) remains an announce-only integration.
